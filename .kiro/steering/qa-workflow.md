# QA 분석 워크플로우

## 개요

피처 분석 요청을 받았을 때 따르는 전체 작업 흐름.
각 단계의 세부 규칙은 개별 steering 파일을 참조한다.

## 작업 순서

```
1. 입력 수집
   │
2. PRD/기획문서 분석 (prd-analysis-rules.md)
   │
3. 피그마 분석 (figma-analysis-rules.md)
   │
3.5. Tech Spec 분석 (tech-spec-analysis-rules.md)
   │
4. 통합 & 이해충돌 검토
   │
5. 통합 명세 파일 생성/업데이트
   │
6. 테스트 케이스 작성 (test-case-rules.md)
   │
7. 최종 검토
```

## 단계별 설명

### 1. 입력 수집

- 제공된 자료 확인: PRD 문서, 피그마 링크, Tech Spec, 추가 컨텍스트
- **Jira 티켓 URL이 제공된 경우**: API로 티켓을 조회하여 디자인닥, 피그마, Tech Spec 링크를 자동 추출
- 누락된 자료가 있으면 요청

#### Jira 티켓 자동 수집

Jira 티켓 URL(`https://{domain}.atlassian.net/browse/{KEY}`)이 제공되면:

1. Jira REST API로 티켓 조회 (`/rest/api/3/issue/{KEY}`)
2. `customfield_10815` 필드에서 링크 추출:
   - `inlineCard` 타입 노드의 `attrs.url`에서 Confluence/Figma URL 추출
   - `link` mark가 있는 텍스트 노드에서 URL 추출
3. 추출된 링크를 역할별로 분류:
   - Confluence URL (`/wiki/`) → PRD(디자인닥) 또는 Tech Spec
   - Figma URL (`figma.com/design/`) → 피그마 분석 대상
   - 기타 URL → 참고 자료
4. 티켓 본문에서 QA 유의사항, 작업 내용, 영향 범위 텍스트도 추출

**인증**: `.env` 파일의 `CONFLUENCE_EMAIL` + `CONFLUENCE_TOKEN`을 Jira API 인증에도 동일하게 사용 (같은 Atlassian 도메인)

#### Jira QA Epic 자동 생성

QA 요청 티켓(이슈타입: `FQA-QA요청`)으로 분석을 시작하는 경우, **분석 시작 전에** QA Epic을 먼저 생성한다:

1. **Epic 생성 + Parent 연결** — 배치 스크립트 1회 실행으로 처리:
   ```bash
   ./scripts/jira-create-epic.sh <QA요청KEY> "<Epic_Summary>" "<디자인닥URL>" "<피그마URL>" [TechSpecURL...]
   ```
   - Project: QA요청 티켓과 동일 프로젝트 (FQA, id: 11206)
   - Issue Type: Epic (id: 10004)
   - Summary: `[QA] {피처명}` (QA요청 티켓의 summary에서 피처명 추출)
   - Description: 디자인닥, 피그마, Tech Spec, QA요청 티켓 링크를 `inlineCard`로 포함
   - QA요청 티켓의 Parent를 자동으로 새 Epic으로 연결
   - 생성된 Epic KEY는 `.last_epic_key` 파일에 저장됨

2. **이후 분석 진행** — TC 확정 시 Epic에 TC HTML 첨부

#### TC 첨부 규칙

TC가 확정(셀프 체크 통과)되면:

1. **TC HTML 생성** — 로컬 산출물로 생성 (Jira에는 첨부하지 않음)
2. **Google Sheets 시트 추가 여부 질문** — 사용자에게 "스프레드시트에 TC 시트를 추가할까요?" 확인
3. **시트 추가할 경우**:
   - 사용자에게 "어떤 스프레드시트에 추가할까요? (URL)" 질문
   - 시트 추가 시도
   - **권한 실패 시**: Service Account 이메일(`gspread@finda-stg.iam.gserviceaccount.com`)을 알려주며 "이 계정을 스프레드시트 편집자로 초대해주세요" 가이드
   - 권한 확인 후 시트 생성 (통계 + 드롭다운 + 조건부 서식 — `gsheet-output-rules.md` 참조)
4. **시트 추가 후 Jira Epic에 링크 추가** — Epic Description에 스프레드시트 URL을 `inlineCard`로 추가

### 2. PRD/기획문서 분석

- `prd-analysis-rules.md` 규칙에 따라 분석
- 확정/미확정 판별, 최신 버전 기준 적용
- 출력: 추출된 정책 항목 목록

### 3. 피그마 분석

- `figma-analysis-rules.md` 규칙에 따라 분석
- 화면 프레임, 플로우차트, annotations 모두 확인
- 출력: 화면 목록, 상태 전이, UI 구조

### 3.5. Tech Spec 분석

- `tech-spec-analysis-rules.md` 규칙에 따라 분석
- Backend Tech Spec (Jupiter/Earth)과 Frontend Tech Spec (Web)을 구분하여 처리
- 출력: API 엔드포인트 목록, 분기 조건, 에러 코드, 데이터 상태 조건

### 4. 통합 & 이해충돌 검토

- PRD 분석 결과와 피그마 분석 결과를 대조
- **Tech Spec과의 대조도 포함** — API 분기 조건이 PRD/피그마와 일치하는지 확인
- **화면 단위 정책-피그마 대조**:
  - PRD에 정의된 각 화면/분기별 정책을 피그마의 대응 화면과 1:1로 맞춰본다
  - 정책에 명시된 요소(텍스트, 뱃지, 섹션 구분, 버튼 상태 등)가 피그마 화면에 존재하는지 확인
  - 피그마 화면에는 있지만 정책에 언급되지 않은 요소가 있는지 확인 (누락 가능성)
  - 정책에는 있지만 피그마에 반영되지 않은 요소가 있는지 확인 (디자인 미반영 가능성)
- **충돌 검토 항목**:
  - 화면 분기 조건이 일치하는가
  - 텍스트 문구가 일치하는가
  - 상태 전이 / 플로우가 일치하는가
  - 에러 처리 정책이 일치하는가
  - 비즈니스 규칙(정렬, 선택 로직 등)이 일치하는가
  - **API 응답 기반 분기 조건이 PRD/피그마의 화면 분기와 일치하는가** (Tech Spec 제공 시)
  - **Tech Spec의 에러 코드/처리가 PRD의 에러 정책과 일치하는가** (Tech Spec 제공 시)

#### 검토 결과 제시 방식

검토 완료 후, 발견된 항목들을 **인터랙티브 HTML 체크리스트**(`{feature-name}-checklist.html`)로 생성하여 사용자가 브라우저에서 직접 선택/확정할 수 있도록 한다. 사용자가 각 항목을 확정해야 다음 단계(5. 통합 명세 작성)로 진행한다.

**각 항목에는 반드시 출처 위치를 명시한다** — 사용자가 해당 부분을 직접 확인할 수 있도록 PRD 섹션명/피그마 프레임명을 기재한다.

체크리스트 HTML 구조:

```
## 이해충돌 검토 결과

### 충돌 항목
- [ ] #1: PRD에서 할인율형 폐기(취소선)인데, 피그마에 할인율 아이콘 잔존
       📄 PRD: 3-4. 서비스 정책 > ◼︎ 쿠폰 유형 (취소선: "할인율형: ex. 0.5% 할인")
       🎨 Figma: "0704_사용 가능한 쿠폰O" 프레임 > 할인율 아이콘 잔존
       → PRD 기준 / 피그마 기준 / 확인 필요

- [ ] #2: PRD에 CTA 미노출 정책인데, 피그마에 비활성 버튼 표시
       📄 PRD: 3-5. 화면 정책 > 하단 고정 메인 CTA > CTA 상태값 ("사용 가능 쿠폰 0개인 경우 CTA 미노출")
       🎨 Figma: "0704_사용 가능한 쿠폰X" 프레임 > 비활성 버튼 존재
       → PRD 기준 / 피그마 기준 / 확인 필요

### 미정의 항목 (PRD·피그마 모두 정의 없음)
- [ ] #3: 스낵바 자동 사라짐 시간
       📄 PRD: 언급 없음
       🎨 Figma: "0704_사용 가능한 쿠폰O_다른쿠폰선택" 프레임 > Snackbar 컴포넌트 존재하나 duration 미표기
       → 정책 확인 필요 / TC에서 제외

### 일치 확인 (충돌 없음)
- [x] 헤더 텍스트 분기 — PRD 3-5 화면 정책 · Figma 헤더 영역 일치
- [x] CTA 문구 "쿠폰 적용하고 대출 찾기" — PRD 3-5 · Figma Button/CTA 일치
- [x] 정렬 기준 — PRD 3-5 쿠폰 정렬 기준 · Figma 리스트 순서 일치
```

#### 출처 위치 명시 규칙

| 출처 | 명시 형식 | 예시 |
|------|----------|------|
| PRD | `📄 PRD: {섹션 번호}. {섹션명} > {하위 항목} ({관련 텍스트 인용})` | `📄 PRD: 3-4. 서비스 정책 > ◼︎ 쿠폰 유형 (취소선: "할인율형")` |
| Figma | `🎨 Figma: "{프레임명}" > {해당 요소 설명}` | `🎨 Figma: "0704_사용 가능한 쿠폰O" > 최대혜택 뱃지` |
| Tech Spec | `🔧 Tech Spec ({팀명}): {섹션} > {항목}` | `🔧 Tech Spec (Earth): 9. 결과 화면 분기 > rewardItemType = VOUCHER` |
| 없음 | `📄 PRD: 언급 없음` 또는 `🎨 Figma: 해당 화면 없음` | — |

- PRD 위치는 문서 내 섹션 번호 + 제목으로 특정한다
- Figma 위치는 프레임명(레이어 이름)으로 특정한다
- 일치 항목도 간략하게 양쪽 위치를 병기한다

사용자가 각 항목에 대해 선택(PRD 기준 / 피그마 기준 / 확인 필요 / TC에서 제외)하면 그 결정을 기반으로 통합 명세와 TC를 작성한다.

- **사용자 확정 전까지 통합 명세 및 TC 작성을 진행하지 않는다.**
- 확정된 결과는 통합 명세의 "미결/확인 필요 사항" 섹션에 이력으로 남긴다.
- **우선순위**: PRD 확정 정책 > 피그마 디자인 (디자인이 아직 업데이트 안 된 경우 있음)

### 5. 통합 명세 파일 생성/업데이트

- 위치: `aidlc-docs/features/{feature-name}.md`
- PRD + 피그마 분석 결과를 하나의 문서로 병합
- 미결 사항 섹션에 충돌/미정의 항목 정리

### 6. 테스트 케이스 작성

- `test-case-rules.md` 규칙에 따라 작성
- 통합 명세 파일을 근거로 TC 도출
- md와 html 동시 생성/수정
- 근거 없는 추측 TC 금지

### 6.5. TC 셀프 체크 (멀티 리뷰어)

TC 생성 후 3가지 관점의 리뷰어가 순차로 검증한다.

#### 리뷰어 역할

| 리뷰어 | 관점 | 체크 포인트 |
|--------|------|------------|
| **QA 리드** | 커버리지 & 구조 | 커버리지 매트릭스 기반 빈 셀 감지, 분기 양쪽 경로 TC 존재, 진입 경로 분리, 네비게이션 TC 포함, 고정 문구 TC 포함, TC 간 중복 없음, 1 case = 1 결과 원칙, **동일 UI 다중 트리거 시 종료 후 동작이 트리거별로 맞는지 확인**, **통합 명세의 확정 이력/미결 사항에 "TC 포함" 결정이 있는 항목이 모두 실제 TC로 존재하는지 1:1 대조** |

##### QA 리드 — 커버리지 매트릭스

QA 리드 리뷰 시 먼저 **화면 × 상태 × 액션** 매트릭스를 생성하고, 각 셀에 대응하는 TC가 있는지 확인한다.

매트릭스 축:

| 축 | 값 도출 기준 |
|----|-------------|
| 화면 | 피그마 프레임 목록 (각 상태별 화면) |
| 상태 | 정상 / 에러 / 빈 상태 / 로딩 (해당 화면에 존재하는 상태만) |
| 액션 | 정상 동작 확인 / 네비게이션(뒤로가기·닫기) / CTA 클릭 / 선택 변경 (해당 화면의 인터랙션 요소 기반) |

매트릭스 셀 판정:

| 표시 | 의미 |
|------|------|
| ✓ | 대응 TC 존재 |
| ✗ | 대응 TC 없음 → 누락 후보 |
| — | 해당 조합이 존재하지 않음 (예: 빈 상태가 없는 화면) |

- `✗` 셀이 있으면 누락 TC로 보고한다.
- 매트릭스는 리뷰 결과에 포함하여 사용자가 한눈에 커버리지를 파악할 수 있게 한다.

예시:

```
[QA 리드 리뷰 — 커버리지 매트릭스]

| 화면              | 정상 | 에러 | 빈 상태 | 네비게이션 |
|-------------------|------|------|---------|-----------|
| 프로모션 페이지    | ✓    | ✓    | —       | ✓          |
| 동의서 바텀싯      | ✓    | —    | —       | —          |
| 로딩 페이지        | ✓    | —    | —       | ✓          |
| 오류 팝업          | —    | ✓    | —       | ✓          |
| 결과 (당첨)        | ✓    | —    | —       | ✓          |
| 결과 (포인트)      | ✓    | —    | —       | ✓          |

누락 없음 ✓
```

##### QA 리드 — 확정 이력 대조 (필수)

커버리지 매트릭스와 별개로, **통합 명세의 "미결/확인 필요 사항 > 확정 이력"과 "PRD 업데이트 반영 이력"을 스캔**하여 "TC 포함", "TC 작성", "TC 추가" 등 TC 생성을 약속한 결정사항을 모두 추출한다. 각 결정사항이 실제 TC로 존재하는지 TC_ID를 명시하여 1:1 대조한다.

```
[QA 리드 리뷰 — 확정 이력 대조]

| 확정 이력 항목 | 결정 | 대응 TC | 상태 |
|---------------|------|---------|------|
| #3 동의서 상세 화살표 클릭 시 이동 | 약관 상세 페이지 이동 TC 포함 | TC-01-12, 02-11, 03-10, 04-10 | ✓ |

누락 없음 ✓
```

- 결정사항에 대응하는 TC가 없으면 **누락 TC로 보고하고 자동 추가**한다 (커버리지 매트릭스 빈 셀과 동일 처리).
- "TC에서 제외", "SCOPE OUT", "미정의" 결정은 대조 대상에서 제외한다.

| **PO** | 비즈니스 정합성 | TC가 정책 의도를 정확히 반영하는가, 우선순위가 비즈니스 임팩트와 일치하는가, 핵심 플로우에 빠진 시나리오가 없는가, 사용자 관점에서 의미 있는 검증인가 |

#### 리뷰 수행 방식

각 리뷰어는 다음 형식으로 피드백을 생성한다:

```
[QA 리드 리뷰]
✓ 화면 커버리지: 6/6 프레임
✓ 분기 양쪽 경로: 5/5
⚠ TC-04-04: 기대 결과 2개 포함 — 분리 검토 필요

[개발자 리뷰]
✓ 사전 조건 재현 가능성: 전체 통과
⚠ TC-10-01: "DB 저장" 검증 방법 명확화 필요 (직접 DB 조회 vs API 응답 확인)

[PO 리뷰]
✓ 핵심 플로우 커버: 전체 통과
✓ 우선순위 정합성: 일치
```

#### 검증 결과 처리

```
리뷰어 3명 순차 검증
  │
  ▼
피드백 합산
  │
  ├─ 위반 0건 → TC 확정. "✅ 셀프 체크 통과 (N건, 리뷰어 3명 합의)" 보고 후 완료
  │
  └─ 위반 있음 → 위반 항목 통합 보고
                  → 자동 수정 가능 항목 (용어 위반, 누락 TC 추가): 에이전트가 수정
                  → 판단 필요 항목 (분리 여부, 검증 방법): 사용자에게 확인
                  → 사용자 확인 후 TC 확정
```

- 자동 수정 가능: 용어 통일 위반, 명백한 TC 누락 (분기 한쪽만 있는 경우), 네비게이션 TC 누락
- 사용자 확인 필요: 1 case = 1 결과 위반 (의도적 병합일 수 있음), 검증 방법 선택, 우선순위 조정 제안
- 셀프 체크 결과는 TC HTML 하단에 접을 수 있는 섹션으로 기록한다.

### 7. 최종 검토

- TC의 모든 기대 결과가 통합 명세에 근거가 있는지 확인
- 미정의 항목이 올바르게 태그되었는지 확인
- md/html 동기화 확인

## 산출물 규칙

- 분석 내용은 대화창에 길게 쓰지 않는다. 파일로 산출물을 생성한다.
- 이해충돌 검토 결과는 **인터랙티브 HTML 체크리스트**(`aidlc-docs/features/{feature-name}-checklist.html`)로 생성하여 사용자가 브라우저에서 직접 선택/확정할 수 있도록 한다.
- 체크리스트 HTML은 로컬 스토리지를 활용하여 선택 상태를 저장한다.
- 대화창에는 "파일 생성 완료" + 간단한 요약(3줄 이내)만 출력한다.
- 모든 산출물은 `aidlc-docs/features/` 하위에 피처별로 분리한다.

### 피처 메타 정보 (`meta.json`)

각 피처 폴더에 `meta.json`을 관리하여 Jira Epic, 스프레드시트 URL 등을 저장한다. TC 업데이트/스프레드시트 반영 시 이 파일을 참조하여 사용자에게 재질문하지 않는다.

```json
{
  "featureName": "feature-name",
  "featureNameKo": "피처 한글명",
  "jiraEpicKey": "FQA-123",
  "jiraEpicUrl": "https://findainc.atlassian.net/browse/FQA-123",
  "spreadsheetUrl": "https://docs.google.com/spreadsheets/d/...",
  "spreadsheetSheetName": "3차",
  "createdAt": "YYYY-MM-DD",
  "updatedAt": "YYYY-MM-DD"
}
```

- Epic 생성 시 `jiraEpicKey`, `jiraEpicUrl` 저장
- 스프레드시트 시트 추가 시 `spreadsheetUrl`, `spreadsheetSheetName` 저장
- TC 업데이트 시 `updatedAt` 갱신
- 값이 비어있으면 해당 작업이 아직 수행되지 않은 것으로 판단 → 사용자에게 질문

### 문서 일괄 업데이트 시 누락 방지

워크플로우나 규칙이 변경되어 여러 문서를 동시에 업데이트해야 할 때:

1. **변경 키워드로 전수 검색** — `grep_search`로 관련 키워드(예: "TC 첨부", "Epic", "워크플로우", 변경 전 문구)를 검색하여 해당 내용이 존재하는 모든 파일 + 위치를 먼저 파악한다.
2. **변경 대상 목록 작성** — 검색 결과를 기반으로 수정할 파일:위치 목록을 만든다.
3. **목록 기반으로 순차 수정** — 목록의 모든 항목을 처리한 후에만 완료로 보고한다.
4. **수정 후 재검색** — 변경 전 문구로 다시 검색하여 잔존 여부를 확인한다.

## 부분 작업 시

- 피그마만 제공된 경우: 2단계 건너뛰고 3→4→5→6 진행 (PRD 없이 분석 가능한 범위만)
- PRD만 제공된 경우: 3단계 건너뛰고 2→4→5→6 진행
- Tech Spec이 제공된 경우: 3.5단계 수행 후 결과를 통합 명세의 "기술 요구사항" 섹션과 TC의 사전 조건에 반영
- Tech Spec만 추가 제공된 경우 (기존 명세 있을 때): 3.5→4→5→6 진행 (기존 TC에 API 정보 보강)
- 추가 자료가 나중에 제공된 경우: 해당 단계 수행 후 4단계(이해충돌 검토)부터 재실행

## 업데이트 시

- 기존 명세가 있는 상태에서 새 자료가 추가되면:
  1. 새 자료 분석
  2. 기존 명세 + PRD + 피그마 전체와 이해충돌 검토
  3. 명세 업데이트
  4. TC 업데이트 (md + html 동시)

### 체크리스트 확정 후 추가 정책 변경 시

이해충돌 체크리스트 확정 + TC 생성 완료 이후에 추가 정책 변경이 들어오는 경우:

1. **체크리스트 재생성 불필요** — 사용자가 직접 전달한 확정 내용이므로 별도 체크리스트 없이 바로 반영한다.
2. **통합 명세 업데이트** — 해당 화면 정책 테이블을 직접 수정한다.
3. **변경 이력 테이블 추가** — 통합 명세의 "미결/확인 필요 사항" 섹션 하단에 `### PRD 업데이트 반영 이력 (날짜)` 테이블을 추가한다:
   ```markdown
   ### PRD 업데이트 반영 이력 (YYYY.MM.DD)

   | 변경 사항 | 이전 | 이후 |
   |-----------|------|------|
   | 항목 설명 | 변경 전 내용 | 변경 후 내용 |
   ```
4. **TC 동시 업데이트** — 변경된 정책에 영향받는 TC를 수정/삭제/추가한다 (md + html 동시).
5. **TC 메타 정보 갱신** — TC 파일 상단에 `PRD 업데이트 반영: YYYY-MM-DD` 표기를 추가/갱신한다.
6. **커버리지 요약 갱신** — TC 수가 변경되면 하단 커버리지 요약 테이블도 업데이트한다.

## 피드백 로그 운영

사용자로부터 TC에 대한 피드백을 받으면 다음 프로세스를 수행한다.

### 피드백 수신 시

1. **TC 즉시 수정** — 피드백 내용을 TC에 반영한다 (md + html 동시).
2. **피드백 로그 기록** — `.kiro/steering/feedback-log.md`에 피드백을 누적한다.
3. **규칙 승격 판단** — 피드백이 범용 적용 가능하면 즉시 steering 규칙에 반영한다.

### 규칙 승격 기준

| 조건 | 처리 |
|------|------|
| 모든 피처에 범용 적용 가능 + 명확한 규칙화 가능 | 즉시 승격 → steering 규칙 추가 |
| 같은 유형 피드백 3건 이상 누적 | 상위 규칙으로 일반화 후 승격 |
| 특정 피처에만 해당 또는 과잉 TC 우려 | 보류 → 잠재 패턴으로 기록 |

### 피드백 로그 참조

- TC 생성 시 `feedback-log.md`의 기존 피드백을 참고하여 같은 실수를 반복하지 않는다.
- 특히 "패턴 분석 > 승격된 규칙" 목록을 TC 생성 전에 확인한다.
