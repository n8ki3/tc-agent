# QA 분석 워크플로우

## 개요

피처 분석 요청을 받았을 때 따르는 전체 작업 흐름.
각 단계의 세부 규칙은 개별 steering 파일을 참조한다.

## 작업 순서

```
1. 입력 수집
   │
2. PRD/기획문서 분석 (prd-analysis-rules.md)
   │
3. 피그마 분석 (figma-analysis-rules.md)
   │
4. 통합 & 이해충돌 검토
   │
5. 통합 명세 파일 생성/업데이트
   │
6. 테스트 케이스 작성 (test-case-rules.md)
   │
7. 최종 검토
```

## 단계별 설명

### 1. 입력 수집

- 제공된 자료 확인: PRD 문서, 피그마 링크, 추가 컨텍스트
- 누락된 자료가 있으면 요청

### 2. PRD/기획문서 분석

- `prd-analysis-rules.md` 규칙에 따라 분석
- 확정/미확정 판별, 최신 버전 기준 적용
- 출력: 추출된 정책 항목 목록

### 3. 피그마 분석

- `figma-analysis-rules.md` 규칙에 따라 분석
- 화면 프레임, 플로우차트, annotations 모두 확인
- 출력: 화면 목록, 상태 전이, UI 구조

### 4. 통합 & 이해충돌 검토

- PRD 분석 결과와 피그마 분석 결과를 대조
- **화면 단위 정책-피그마 대조**:
  - PRD에 정의된 각 화면/분기별 정책을 피그마의 대응 화면과 1:1로 맞춰본다
  - 정책에 명시된 요소(텍스트, 뱃지, 섹션 구분, 버튼 상태 등)가 피그마 화면에 존재하는지 확인
  - 피그마 화면에는 있지만 정책에 언급되지 않은 요소가 있는지 확인 (누락 가능성)
  - 정책에는 있지만 피그마에 반영되지 않은 요소가 있는지 확인 (디자인 미반영 가능성)
- **충돌 검토 항목**:
  - 화면 분기 조건이 일치하는가
  - 텍스트 문구가 일치하는가
  - 상태 전이 / 플로우가 일치하는가
  - 에러 처리 정책이 일치하는가
  - 비즈니스 규칙(정렬, 선택 로직 등)이 일치하는가

#### 검토 결과 제시 방식

검토 완료 후, 발견된 항목들을 **인터랙티브 HTML 체크리스트**(`{feature-name}-checklist.html`)로 생성하여 사용자가 브라우저에서 직접 선택/확정할 수 있도록 한다. 사용자가 각 항목을 확정해야 다음 단계(5. 통합 명세 작성)로 진행한다.

**각 항목에는 반드시 출처 위치를 명시한다** — 사용자가 해당 부분을 직접 확인할 수 있도록 PRD 섹션명/피그마 프레임명을 기재한다.

체크리스트 HTML 구조:

```
## 이해충돌 검토 결과

### 충돌 항목
- [ ] #1: PRD에서 할인율형 폐기(취소선)인데, 피그마에 할인율 아이콘 잔존
       📄 PRD: 3-4. 서비스 정책 > ◼︎ 쿠폰 유형 (취소선: "할인율형: ex. 0.5% 할인")
       🎨 Figma: "0704_사용 가능한 쿠폰O" 프레임 > 할인율 아이콘 잔존
       → PRD 기준 / 피그마 기준 / 확인 필요

- [ ] #2: PRD에 CTA 미노출 정책인데, 피그마에 비활성 버튼 표시
       📄 PRD: 3-5. 화면 정책 > 하단 고정 메인 CTA > CTA 상태값 ("사용 가능 쿠폰 0개인 경우 CTA 미노출")
       🎨 Figma: "0704_사용 가능한 쿠폰X" 프레임 > 비활성 버튼 존재
       → PRD 기준 / 피그마 기준 / 확인 필요

### 미정의 항목 (PRD·피그마 모두 정의 없음)
- [ ] #3: 스낵바 자동 사라짐 시간
       📄 PRD: 언급 없음
       🎨 Figma: "0704_사용 가능한 쿠폰O_다른쿠폰선택" 프레임 > Snackbar 컴포넌트 존재하나 duration 미표기
       → 정책 확인 필요 / TC에서 제외

### 일치 확인 (충돌 없음)
- [x] 헤더 텍스트 분기 — PRD 3-5 화면 정책 · Figma 헤더 영역 일치
- [x] CTA 문구 "쿠폰 적용하고 대출 찾기" — PRD 3-5 · Figma Button/CTA 일치
- [x] 정렬 기준 — PRD 3-5 쿠폰 정렬 기준 · Figma 리스트 순서 일치
```

#### 출처 위치 명시 규칙

| 출처 | 명시 형식 | 예시 |
|------|----------|------|
| PRD | `📄 PRD: {섹션 번호}. {섹션명} > {하위 항목} ({관련 텍스트 인용})` | `📄 PRD: 3-4. 서비스 정책 > ◼︎ 쿠폰 유형 (취소선: "할인율형")` |
| Figma | `🎨 Figma: "{프레임명}" > {해당 요소 설명}` | `🎨 Figma: "0704_사용 가능한 쿠폰O" > 최대혜택 뱃지` |
| 없음 | `📄 PRD: 언급 없음` 또는 `🎨 Figma: 해당 화면 없음` | — |

- PRD 위치는 문서 내 섹션 번호 + 제목으로 특정한다
- Figma 위치는 프레임명(레이어 이름)으로 특정한다
- 일치 항목도 간략하게 양쪽 위치를 병기한다

사용자가 각 항목에 대해 선택(PRD 기준 / 피그마 기준 / 확인 필요 / TC에서 제외)하면 그 결정을 기반으로 통합 명세와 TC를 작성한다.

- **사용자 확정 전까지 통합 명세 및 TC 작성을 진행하지 않는다.**
- 확정된 결과는 통합 명세의 "미결/확인 필요 사항" 섹션에 이력으로 남긴다.
- **우선순위**: PRD 확정 정책 > 피그마 디자인 (디자인이 아직 업데이트 안 된 경우 있음)

### 5. 통합 명세 파일 생성/업데이트

- 위치: `aidlc-docs/features/{feature-name}.md`
- PRD + 피그마 분석 결과를 하나의 문서로 병합
- 미결 사항 섹션에 충돌/미정의 항목 정리

### 6. 테스트 케이스 작성

- `test-case-rules.md` 규칙에 따라 작성
- 통합 명세 파일을 근거로 TC 도출
- md와 html 동시 생성/수정
- 근거 없는 추측 TC 금지

### 7. 최종 검토

- TC의 모든 기대 결과가 통합 명세에 근거가 있는지 확인
- 미정의 항목이 올바르게 태그되었는지 확인
- md/html 동기화 확인

## 산출물 규칙

- 분석 내용은 대화창에 길게 쓰지 않는다. 파일로 산출물을 생성한다.
- 이해충돌 검토 결과는 **인터랙티브 HTML 체크리스트**(`aidlc-docs/features/{feature-name}-checklist.html`)로 생성하여 사용자가 브라우저에서 직접 선택/확정할 수 있도록 한다.
- 체크리스트 HTML은 로컬 스토리지를 활용하여 선택 상태를 저장한다.
- 대화창에는 "파일 생성 완료" + 간단한 요약(3줄 이내)만 출력한다.
- 모든 산출물은 `aidlc-docs/features/` 하위에 피처별로 분리한다.

## 부분 작업 시

- 피그마만 제공된 경우: 2단계 건너뛰고 3→4→5→6 진행 (PRD 없이 분석 가능한 범위만)
- PRD만 제공된 경우: 3단계 건너뛰고 2→4→5→6 진행
- 추가 자료가 나중에 제공된 경우: 해당 단계 수행 후 4단계(이해충돌 검토)부터 재실행

## 업데이트 시

- 기존 명세가 있는 상태에서 새 자료가 추가되면:
  1. 새 자료 분석
  2. 기존 명세 + PRD + 피그마 전체와 이해충돌 검토
  3. 명세 업데이트
  4. TC 업데이트 (md + html 동시)
