# 테스트 케이스 작성 규칙

## 기본 원칙

- 테스트 케이스의 기대 결과는 **사용자가 화면에서 텍스트로 인지할 수 있는 요소**로 작성한다.
- 디자인 구현 디테일(px, 색상코드, 컴포넌트 크기, 좌표 등)은 포함하지 않는다.
- "시각적으로 구분된다" 수준의 표현은 허용하되, 구체적 디자인 속성은 제외한다.
- **기대 결과는 반드시 PRD, 기획서, 피그마 등 명시된 근거가 있어야 한다. 추측으로 작성하지 않는다.**

## 근거 기반 작성 원칙

- 기대 결과의 모든 항목은 PRD, 기획서, 피그마, 또는 확정된 정책 문서에서 확인 가능해야 한다.
- 명시되지 않은 동작은 기대 결과에 포함하지 않는다.
- 일반적인 관행이나 추측으로 기대 결과를 채우지 않는다.
- 명세에 정의되지 않았지만 검증이 필요한 동작은 "확인 필요" 또는 "미정의"로 표기하고, 미결 사항으로 분리한다.

### 근거로 인정되는 것

- PRD/기획서에 텍스트로 명시된 정책
- 피그마 화면 프레임에 표시된 UI 요소 (텍스트, 버튼, 상태)
- 피그마 플로우차트의 분기 조건과 화살표로 연결된 이동 경로
- 피그마 핸드오프 노트에 기재된 설명
- 이벤트 로그 정의서에 명시된 이벤트명과 파라미터
- 확정된 회의록/논의 결과
- **피그마 화면에 존재하는 네비게이션 요소** (뒤로가기 버튼, 닫기 버튼 등 — 존재 자체가 근거)

### 근거로 인정되지 않는 것 (추측)

- "일반적으로 이렇게 동작할 것이다"라는 업계 관행
- 유사 서비스의 동작 방식 유추
- 기술적 구현 방식에 대한 가정 (예: 디퍼드 딥링크 지원 여부)
- 명시되지 않은 에러/예외 케이스의 처리 방식
- 명시되지 않은 UI 인터랙션 세부 동작 (애니메이션, 타이밍 등)

### TC 분리 기준

- 명세에 정의된 에러 정책이 하나의 케이스로 커버되면 원인별로 TC를 분리하지 않는다. (예: "API 호출 실패" 정책이 있으면 네트워크 끊김, 서버 에러, 타임아웃 등을 별도 TC로 만들지 않음 — 결국 동일한 에러 처리로 동작하므로)
- TC를 분리하는 기준은 **명세에서 다른 처리 정책이 정의된 경우**에 한한다.
- **진입 경로가 물리적으로 다른 위치(화면/채널)에서 시작되면 각각 별도 TC로 분리한다.** (예: CRM PUSH 진입과 스마트배너 진입은 유입 위치가 다르므로 별도 TC. 동일 화면 내 버튼 2개는 위치가 같으므로 분리 불필요)
- **선택 항목이 후속 동작에 영향을 주는 경우, 선택/미선택 양쪽 케이스를 모두 작성한다.** (예: 광고성 수신 동의 체크 시 Snackbar 노출 → 미체크 시 Snackbar 미노출도 별도 TC로 작성)

## 기대 결과 작성 기준

### 포함해야 할 것 (사용자 인지 가능 요소)

- 화면에 표시되는 텍스트 (헤더, 버튼 문구, 안내 메시지, 토스트 등)
- 요소의 노출/미노출 여부
- 선택/비선택 상태 변화 (강조 표시 여부)
- 화면 이동 (랜딩 페이지)
- 이벤트 로그 발생 여부 및 파라미터
- 리스트 순서
- 개수 표기
- 활성/비활성 상태

### 제외해야 할 것 (디자인 구현 디테일)

- 컴포넌트 높이/너비 (px)
- 색상 코드 (#hex, rgb)
- 좌표 (x, y)
- 폰트 크기/굵기
- 아이콘 파일명 (내부 asset명)
- 패딩/마진 값
- 애니메이션 duration

## 예시

### ❌ Bad

```
• A 해제 (97px 일반 스타일)
• B 선택 (115px 강조 스타일)
• 스낵바 노출 (321×52, CTA 위)
• 정액형: iic_benefit_coupon_point_01 아이콘
```

### ✅ Good

```
• A 선택 해제됨 (강조 표시 사라짐)
• B 선택됨 (강조 표시)
• 하단에 스낵바 메시지 노출
• 정액형 쿠폰: 포인트 아이콘 표시
```

## 공통 검증 패턴 (화면 존재 기반 TC)

피그마 화면에 존재하는 요소는 PRD에 별도 정책이 없더라도 TC 생성 대상이다. 아래 패턴에 해당하면 자동으로 TC를 포함한다.

### 네비게이션 (뒤로가기)

| 조건 | TC 생성 | 기대 결과 | 우선순위 |
|------|---------|----------|---------|
| Header에 뒤로가기 버튼 존재 + 트랜지션 명시 없음 | 진입 경로별 "이전 화면 이동" TC | 이전 화면으로 이동 | Medium |
| Header에 뒤로가기 버튼 존재 + 트랜지션 명시 있음 | 명시된 목적지로 이동 TC | 해당 화면으로 이동 | Medium |
| 뒤로가기가 유일한 이탈 경로 | 위와 동일하되 우선순위 상향 | — | High |

### 고정 문구 / 법적 고지

| 조건 | TC 생성 | 기대 결과 | 우선순위 |
|------|---------|----------|---------|
| 화면에 안내사항/유의사항/법적 고지 영역(Disclaimer 등) 존재 | "안내사항 노출 확인" TC 1건 | 안내사항 영역 노출됨 | Low |

- 문구 전문을 기대 결과에 넣지 않는다. 문구는 수시로 변경되며, 테스트 시 피그마와 대조 확인한다.
- 기대 결과는 **"안내사항 영역 노출됨"** 수준으로 작성한다.
- 노출 여부를 놓치지 않고 TC에 포함하는 것이 목적이다.

### BO (백오피스)

| 조건 | TC 생성 | 기대 결과 | 우선순위 |
|------|---------|----------|---------|
| PRD에 BO 요구사항 존재 (필드 추가/수정/삭제) | 별도 카테고리 "BO-XX"로 TC 생성 | 필드 노출, 유효성 검증, 앱 반영 확인 | Medium |

- BO TC는 PRD에 명시된 변경 사항만 대상으로 한다 (기존 기능 재검증 제외).
- 카테고리 ID는 `BO-01`, `BO-02` 형식으로 분리한다.

## 우선순위 판단 기준

| 우선순위 | 기준 | 예시 |
|---------|------|------|
| **High** | 우회 불가능한 핵심 플로우의 진행 불가 | CTA 클릭 시 다음 화면 미이동, 데이터 미저장, 결제/약정 연결 실패 |
| **Medium** | 명시된 기능의 오동작, 정책 위반 | 에러 팝업 문구 불일치, 정렬 순서 오류, 이벤트 로그 미발생, 분기 조건 오동작 |
| **Low** | 부가 기능, 편의성, UI/UX | 토스트 노출 조건, 동의서 상세 이동, 하단 안내 문구 표시 |

- TC HTML 뱃지 색상: High(빨강 `#dc2626`), Medium(주황 `#d97706`), Low(파랑 `#2563eb`)

## 구조

각 테스트 케이스는 다음 컬럼을 포함한다:

| 컬럼 | 설명 |
|------|------|
| TC_ID | 카테고리-순번 형식 (예: TC-01-01) |
| Category1 | 화면 Depth 1단계 (최상위 화면/탭) |
| Category2 | 화면 Depth 2단계 (하위 화면/섹션) |
| Category3 | 화면 Depth 3단계 (세부 영역, 필요 시) |
| Level | High / Medium / Low (우선순위 판단 기준 참조) |
| pre-Condition | 테스트 실행 전 필요한 데이터/상태, 테스트 데이터 포함 |
| Test Procedure | 해당 화면 Depth까지 이동한 이후의 동작만 기입 (화면 이동 경로는 Category로 표현됨) |
| Expected Result | 사용자가 인지할 수 있는 결과 (기대 결과 작성 기준 준수) |
| Android | Android 수행 결과 (PASS / FAIL / N/T / N/A) |
| iOS | iOS 수행 결과 (PASS / FAIL / N/T / N/A) |
| Comment | 특이/공유사항, N/T·N/A 사유, 이슈 링크 |

### Category (화면 Depth) 작성 규칙

- Category1~3은 화면의 하이어라키 또는 연속적인 화면의 깊이를 나타낸다.
- 예: Category1=소식, Category2=쿠폰함, Category3=만료된 쿠폰
- 기본 템플릿은 Category3까지 구성하며, 필요한 경우에만 추가한다.

### Test Procedure 작성 규칙

- 화면 Depth까지의 이동 경로는 작성하지 않는다 (Category가 이미 표현).
- 해당 화면에 도달한 이후의 **동작만** 기입한다.
- 기대결과로 이어지는 수행절차는 동작 목적까지 작성한다 (예: "노출 확인", "동작 확인").

### Expected Result 작성 규칙

- 말줄임 방식으로 작성한다 (예: "시트 노출", "페이지 이동").
- 화면 UI를 확인하는 case는 가능한 한 화면 구성을 표현한다.
- 하나의 case는 하나의 기대결과를 가진다.

### 수행 결과 값

| 값 | 의미 |
|----|------|
| PASS | 정상 동작 확인 |
| FAIL | 결함 발견 |
| N/T | 수행 가능하지만 이번 검증에서 제외 |
| N/A | 해당 OS에서 수행 불가능하거나 검증 시점에 수행 불가 |

## 용어 통일 규칙

TC 작성 시 다음 용어를 통일하여 사용한다:

| 기존 표현 | 통일 표현 |
|-----------|----------|
| 클릭, 터치 | **선택** |
| 랜딩, 진입, 전환 | **이동** |
| 노출, 출력, 표시 | **노출** |
| 바텀시트 | **바텀싯** |
| close, 닫기, 종료 (Test Procedure) | **닫기** |
| close, 닫기, 종료 (Expected Result) | **닫힘** |
| expand, 확장/축소 | **펼치기/접기** |
| intro | **인트로** |
| 미니 인트로 | **레드카펫** |

### 표기 규칙

- 버튼: `[버튼명]` 으로 감싼다 (예: `[쿠폰 적용하고 대출 찾기]`)
- 버튼이 아닌 텍스트 요소: `"텍스트"` 로 감싼다 (예: `"최대 혜택 쿠폰을 자동으로 선택했어요"`)
- 앱에 노출되는 label을 사용한다 (사내 용어/실무 용어 사용 금지)
- 타인이 수행 가능하도록 자세히 작성한다

## TC History 관리

TC 마크다운 상단에 TC History 테이블을 포함한다:

```markdown
| Date | Action | TC_ID | Writer |
|------|--------|-------|--------|
| YYYY-MM-DD | 신규 작성 / 추가 / 수정 / 삭제 | 대상 TC_ID 또는 "전체" | 작성자 |
```

- 내용 변경 시마다 행을 추가한다.
- Action: 추가 / 수정 / 삭제
- TC_ID: 변경 대상 TC ID (전체 신규 작성 시 "전체")

## 미정의 항목 처리

- 명세에 정의되지 않았지만 검증이 필요해 보이는 동작은 `[미정의]` 태그를 붙여 기대 결과와 분리한다.
- HTML 출력 시 미정의 항목은 보라색(`#7c3aed`) 텍스트로 시각적 구분한다.
- 미정의 항목은 PO/개발에 확인 요청 후, 확정되면 정식 기대 결과로 승격하거나 삭제한다.

## 출력 형식

### 산출물 목록

피처 분석 시 다음 산출물을 `aidlc-docs/features/` 하위에 생성한다:

| 산출물 | 파일명 | 설명 |
|--------|--------|------|
| 통합 명세 | `{feature-name}.md` | PRD + 피그마 분석 결과를 하나로 병합한 문서 |
| 이해충돌 체크리스트 | `{feature-name}-checklist.html` | PRD↔피그마 충돌/미정의 항목을 사용자가 브라우저에서 확정하는 인터랙티브 HTML |
| TC 마크다운 | `{feature-name}-test-cases.md` | 테스트 케이스 본문 |
| TC HTML | `{feature-name}-test-cases.html` | 브라우저에서 시각적으로 확인 가능한 TC |

### 통합 명세 구조

통합 명세(`{feature-name}.md`)는 다음 섹션을 포함한다:

1. **핵심 목표** — KPI, 목표값
2. **이벤트 스킴** — 대상, 기간, 유입구, 리워드, 예산
3. **User Flow** — 텍스트 기반 플로우차트 (공통 네비게이션 포함)
4. **화면 정책 상세** — 화면별 테이블 (문구, 상태, 분기 조건)
5. **포인트/리워드 지급 정책** — 확률, 금액, 인원 (해당 시)
6. **이벤트 로그** — 이벤트 키, 설명, 파라미터
7. **기술 요구사항** — API, 데이터 저장, 연동 사항
8. **미결/확인 필요 사항** — 확정 이력 + 잔여 미확정 항목 + PRD 업데이트 반영 이력

### 이해충돌 체크리스트 규칙

- 충돌 항목 / 미정의 항목 / 일치 항목으로 섹션 분리한다.
- 각 충돌/미정의 항목에는 **출처 위치**(PRD 섹션명, Figma 프레임명)를 반드시 명시한다.
- 각 항목에 선택 가능한 옵션 버튼을 제공한다 (예: "PRD 기준" / "Figma 기준" / "확인 필요").
- 로컬 스토리지로 선택 상태를 저장하여 브라우저 새로고침 시에도 유지한다.
- 모든 항목 확정 후 결과를 클립보드에 복사할 수 있는 내보내기 기능을 포함한다.
- 스타일: 충돌(빨강 배경), 미정의(보라 배경), 일치(초록 배경)로 시각 구분한다.

### TC 출력 형식

- 마크다운(`.md`)과 HTML(`.html`) 두 형식으로 동시 관리한다.
- **test-cases.md를 생성하거나 수정할 때 반드시 대응하는 test-cases.html도 동일하게 생성/수정한다.**
- HTML은 브라우저에서 열어 시각적으로 확인할 수 있도록 스타일링한다.
- **HTML 스타일 및 구조는 `aidlc-docs/features/car-number-promo/car-number-promo-test-cases.html`을 템플릿으로 참조한다.** 동일한 CSS 클래스, 뱃지 스타일, 테이블 구조를 유지한다.
- 우선순위는 뱃지로 표시: High(빨강 `#dc2626`), Medium(주황 `#d97706`), Low(파랑 `#2563eb`).
- 미정의 항목은 보라색(`#7c3aed`) 텍스트로 시각적 구분한다.
- TC 마크다운 상단에 커버리지 요약 테이블(카테고리별 TC 수, 우선순위별 수)을 포함한다.

#### HTML 상단 대시보드

HTML 최상단에 2줄 카드 대시보드를 배치한다:

- **1줄**: 전체 TC | High | Medium | Low (고정값)
- **2줄 (6칸 한 줄)**: 실행률 | 패스율 | Pass | Fail | N/T | N/A (셀렉트박스 선택에 따라 실시간 업데이트)
- 실행률 = (Pass + Fail) / 전체 TC × 100
- 패스율 = Pass / (Pass + Fail) × 100

#### HTML 테이블 인터랙션

- **Android / iOS 열**: `<select>` 셀렉트박스 (옵션: Pass / Fail / N/T / N/A)
- **Comment 열**: `<textarea>` 수정 가능한 입력 필드
- **행 배경색**: Pass 선택 시 푸른색(`#eff6ff`), Fail 선택 시 붉은색(`#fef2f2`). 같은 행에 Pass와 Fail이 동시에 있으면 **Fail(붉은색) 우선**.
- **로컬 스토리지 저장**: 셀렉트박스 값과 textarea 입력값은 `localStorage`에 저장하여 새로고침 시에도 유지한다. 키 형식: `tc_{TC_ID}_{컬럼인덱스}`

- TC 마크다운 형식: `examples/car-number-promo/car-number-promo-test-cases.md`

## 파일 네이밍 규칙

- 피처명은 **영문 kebab-case**로 작성한다. (예: `coupon-box`, `loan-apply`, `credit-intro`)
- 피처별로 **폴더를 생성**하여 산출물 4종을 한 세트로 관리한다.
- 한글 피처명이 주어지면 의미를 유지하면서 영문 kebab-case로 변환한다.
