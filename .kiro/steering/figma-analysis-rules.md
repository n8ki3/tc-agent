# 피그마 분석 규칙

## 분석 범위

피그마 디자인을 분석할 때 다음 요소들을 **모두** 확인한다:

1. **화면 프레임** — 각 상태별 화면 (정상, 에러, 빈 상태 등)
2. **플로우차트 (Sub Parts/Flowchart)** — 진입 경로, 분기 조건, 화면 간 이동 로직
3. **인터랙션 (Sub Parts/Interaction)** — 조건 분기 라벨 (예/아니오, YES/NO)
4. **연결선 (Sub Parts/Line)** — 화면 간 이동 방향과 조건
5. **핸드오프 노트 (Handoff/Title, Handoff/DetailedFlow)** — 디자이너가 남긴 설명
6. **플로우 화살표 코멘트 (Handoff/FlowArrowComment)** — 화면 간 이동에 붙은 조건/상태 라벨 (예: "에러")
7. **Annotations / 주석** — 화면 위에 붙은 설명 텍스트, 외부 문서 참조 링크

## 분석 순서

1. `get_metadata`로 전체 구조 파악 (섹션, 프레임, 플로우차트, 핸드오프 노드 식별)
2. **Handoff/DetailedFlow, Handoff/FlowArrowComment 노드가 있으면 반드시 `get_design_context`로 내용 확인** — 외부 플로우 참조 링크, 에러 분기 라벨 등 핵심 정보 포함
3. 플로우차트/인터랙션 노드가 있으면 `get_design_context`로 텍스트 내용 추출
4. 화면 프레임은 `get_screenshot`으로 시각적 확인
5. 핸드오프 노트가 있으면 반드시 내용 확인

## 플로우차트 분석 시 확인 사항

- 진입 경로 (어디서 이 화면에 도달하는가)
- 분기 조건 (다이아몬드 형태의 조건 노드)
- 분기별 결과 (예/아니오, 조건 충족/미충족)
- 외부 시스템 연동 (앱스토어, 딥링크, 외부 URL 등)
- 에러/예외 경로
- **FlowArrowComment 라벨** (화살표 위에 붙은 "에러", "성공" 등 상태 표시)
- **DetailedFlow 참조 링크** (다른 피그마 파일/페이지로의 연결 — 후속 플로우 정의 위치)

## 이해 충돌 검토 (필수)

새로운 분석 내용이 추가될 때마다 다음 검토를 반드시 수행한다:

1. **기존 명세 파일과 비교** — 이미 작성된 feature 명세(`aidlc-docs/features/`)의 내용과 충돌하는 부분이 없는지 확인
2. **PRD/기획서와 비교** — 피그마에서 새로 확인된 내용이 PRD 정책과 일치하는지, 상충하는지 확인
3. **충돌 발견 시 처리**:
   - 충돌 내용을 명시적으로 보고 (어떤 문서에서 뭐라고 했는데, 새 분석에서는 뭐가 다른지)
   - 어느 쪽이 최신/확정인지 판단할 수 없으면 "미결 사항"으로 등록
   - 확정된 쪽으로 기존 문서 업데이트
4. **충돌 없음 확인 시** — "기존 명세/PRD와 충돌 없음" 한 줄로 보고

이 검토 없이 새 분석 내용을 명세나 TC에 반영하지 않는다.

```
[진입점]
  │
  ▼
◇ 분기 조건?
  ├─ [조건 A] → 결과 A
  └─ [조건 B] → 결과 B
```

## 출력 형식

플로우차트 분석 결과는 텍스트 기반 플로우로 변환하여 명세에 포함한다:

```
[진입점]
  │
  ▼
◇ 분기 조건?
  ├─ [조건 A] → 결과 A
  └─ [조건 B] → 결과 B
```
