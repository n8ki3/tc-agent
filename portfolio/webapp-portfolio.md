# QA TC Generator — 웹 애플리케이션

> Kiro 에이전트에서 검증된 로직을 웹으로 꺼냈습니다. 브라우저에서 PRD + Figma 넣으면 TC가 나옵니다.

---

## 만든 것

Kiro IDE 에이전트로 돌아가던 QA 분석 로직을 독립 웹 서비스로 확장하는 프로젝트입니다.

에이전트는 잘 동작하지만, Kiro IDE를 설치해야 쓸 수 있습니다.
PO나 다른 QA가 "나도 써보고 싶다"고 할 때 "Kiro 깔아"라고 하기엔 진입 장벽이 높습니다.

그래서 브라우저만 열면 되는 버전을 만들고 있습니다.

---

## 왜 웹으로 꺼내나

| 에이전트 버전의 한계 | 웹 서비스로 해결 |
|-------------------|----------------|
| Kiro IDE 설치 필요 | 브라우저만 있으면 됨 |
| 채팅 기반이라 UX가 선형적 | 단계별 UI로 진행 상황 명확 |
| 결과물이 로컬 파일 | 다운로드 버튼 하나로 md/html/xlsx |
| 혼자만 쓸 수 있음 | URL 공유하면 팀 누구나 사용 |

---

## 동작 방식

```
┌─────────────────────────────────────────────────────┐
│  1. 입력                                             │
│     • PRD PDF 드래그 앤 드롭                          │
│     • Figma 링크 입력                                │
│     • [분석 시작]                                    │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  2. 분석 진행 (실시간 상태)                           │
│     • PRD 분석 중... ✓                              │
│     • Figma 분석 중... ✓                            │
│     • 이해충돌 검토 중... ✓                          │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  3. 이해충돌 체크리스트                               │
│     • 충돌 #1: [PRD 기준 ▼]                         │
│     • 미정의 #2: [TC에서 제외 ▼]                     │
│     • [확정하고 TC 생성]                             │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  4. 결과 & 다운로드                                  │
│     • TC 미리보기 (HTML 렌더링)                      │
│     • [통합 명세 .md] [TC .md] [TC .html] [TC .xlsx] │
└─────────────────────────────────────────────────────┘
```

에이전트 버전과 로직은 동일합니다. UI만 채팅에서 웹으로 바뀐 겁니다.
"사용자 확정 게이트"도 그대로 — 3단계에서 확정 안 하면 4단계로 안 넘어갑니다.

## 기술적으로 신경 쓴 부분

### 에이전트 규칙 → LLM 프롬프트 변환

Kiro Steering 파일이 하던 역할을 백엔드 프롬프트로 옮겼습니다.

| 에이전트 자산 | 웹 서비스에서의 역할 |
|--------------|-------------------|
| `prd-analysis-rules.md` | LLM 시스템 프롬프트로 변환 |
| `figma-analysis-rules.md` | Figma API 호출 + 후처리 규칙 |
| `test-case-rules.md` | TC 생성 프롬프트 + 출력 검증 |
| `qa-workflow.md` | 백엔드 파이프라인 상태 머신 |
| `examples/coupon-box/` | few-shot 예시 + 품질 벤치마크 |

규칙을 프롬프트로 바꾸는 게 핵심이었습니다. 규칙 파일 자체가 이미 "이렇게 해라 / 이렇게 하지 마라" 형식이라 변환이 자연스러웠습니다.

### PDF 파싱

PRD PDF에서 텍스트만 뽑으면 안 됩니다. 취소선이 "폐기된 안"을 의미하니까요.

PyMuPDF로 텍스트 + 표 + 취소선 스타일을 함께 추출합니다.
취소선 텍스트는 `~~삭제됨~~`으로 마킹해서 LLM이 "이건 폐기된 정책"으로 인식하게 합니다.

### Figma API 직접 호출

에이전트에서는 Figma MCP가 알아서 해줬지만, 웹 서비스에서는 REST API를 직접 호출합니다.
메타데이터 → 노드 트리 파싱 → 텍스트 추출 → 스크린샷 렌더링까지 직접 구현했습니다.

### 세션 기반 상태 관리

분석 → 체크리스트 확정 → TC 생성이 비동기로 일어납니다.
사용자가 체크리스트를 확정할 때까지 서버가 기다려야 하니까, 세션 상태 머신으로 관리합니다.

```
ANALYZING → CHECKLIST_PENDING → CONFIRMED → GENERATING → COMPLETED
```

각 상태에서 할 수 있는 API 호출이 정해져 있고, 순서를 건너뛸 수 없습니다.

## 프로젝트 구조

```
webapp/
├── frontend/                    ← Next.js + TailwindCSS
│   └── src/
│       ├── app/                 # 페이지 라우팅
│       └── components/
│           ├── InputStep.tsx    # PRD 업로드 + Figma 링크 입력
│           ├── AnalyzingStep.tsx # 분석 진행 상태 표시
│           ├── ChecklistStep.tsx # 이해충돌 체크리스트
│           └── ResultStep.tsx   # TC 미리보기 + 다운로드
├── backend/                     ← Python FastAPI
│   └── app/
│       ├── routers/
│       │   ├── analyze.py      # 분석 시작
│       │   ├── confirm.py      # 체크리스트 확정
│       │   ├── generate.py     # TC 생성
│       │   └── download.py     # 파일 다운로드
│       └── services/
│           ├── pdf_service.py        # PDF 파싱 (PyMuPDF)
│           ├── figma_service.py      # Figma REST API
│           ├── confluence_service.py # Confluence API
│           ├── llm_service.py        # LLM 호출 (Claude)
│           └── export_service.py     # md/html/xlsx 생성
└── README.md
```

## 기술 스택

| 레이어 | 기술 | 선택 이유 |
|--------|------|----------|
| Frontend | Next.js + TailwindCSS | 단계별 UI에 적합, 빠른 프로토타이핑 |
| Backend | FastAPI | 비동기 처리, 타입 힌트, 자동 문서화 |
| LLM | Claude API | 긴 컨텍스트 처리, 한국어 품질 |
| PDF | PyMuPDF | 텍스트 + 스타일(취소선) 동시 추출 |
| Figma | REST API + OAuth | MCP 없이 독립 동작 |
| Export | openpyxl | xlsx 다운로드 지원 |

| Phase | 상태 | 내용 |
|-------|------|------|

| 5. 확장 | 🔲 예정 | Jira 연동, 회귀 TC 자동 식별, 팀 협업 |

## 차별점

기존 TC 자동 생성 도구들과 비교하면:

| 기존 도구 | 이 서비스 |
|----------|----------|
| PRD 텍스트만 입력 | PRD + Figma 동시 분석 |
| 바로 TC 생성 | 이해충돌 검토 → 사용자 확정 → TC 생성 |
| 추측 포함된 TC | 근거 기반 원칙 + 미정의 항목 분리 |
| 일회성 생성 | 명세 업데이트 시 TC 자동 동기화 |

"PRD만 넣으면 TC 나옵니다" 류의 도구는 많습니다.
하지만 PRD와 디자인이 다를 때 — 실무에서 거의 항상 다릅니다 — 그걸 잡아주는 도구는 없었습니다.
