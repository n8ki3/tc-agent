# Git 작업 규칙

## 시작 전 컨펌 (필수)

- git 도구가 autoApprove로 설정되어 있더라도, **git 작업(커밋/푸시/브랜치 생성/PR)을 시작하기 전에 반드시 사용자에게 컨펌을 받는다.**
- "지금 git 작업을 진행할까요?" 형태로 무엇을 할지(브랜치명, 커밋 범위, 푸시/PR 여부)를 먼저 알리고 동의를 받은 뒤 실행한다.
- 사용자가 명시적으로 "git에 올려줘", "커밋해줘", "푸시해줘"라고 요청한 경우는 그 자체가 컨펌이므로 바로 진행한다.
- 컨펌 없이 선제적으로 커밋/푸시하지 않는다.

## 브랜치 전략

- **파일 수정을 시작하기 전에(작업 착수 시점) 먼저 현재 브랜치가 적절한지 확인한다.** 커밋 직전이 아니라 작업 시작 시점에 확인해야, 엉뚱한 브랜치(특히 이미 머지된 브랜치)에 변경이 쌓이는 것을 막는다.
- 확인 절차:
  1. `git branch --show-current`로 현재 브랜치 확인
  2. 현재 브랜치가 **이미 머지되었거나** 이번 작업과 무관하면 → 아래 "새 브랜치 시작" 수행
  3. 같은 맥락의 이어지는 작업이면 → 현재 브랜치에서 계속
- 새 브랜치 시작 (현재 브랜치가 부적절할 때):
  1. (작업 변경이 이미 있으면) `git stash`로 보관
  2. `git checkout main`
  3. `git pull`
  4. 새 브랜치 생성 (`git checkout -b {브랜치명}`)
  5. (stash 했으면) `git stash pop`으로 복원
  6. 커밋 + 푸시
- 같은 맥락의 추가 작업이면: 기존 브랜치에서 이어서 커밋
- **이미 머지된 브랜치에서는 새 작업을 시작하지 않는다.**

## 브랜치 네이밍

- `feat/{기능명}` — 새 기능, 규칙 추가, TC 생성
- `fix/{수정 대상}` — 버그 수정, TC 수정
- `chore/{작업명}` — 설정 변경, 파일 정리, 문서만 수정

## 커밋 규칙

- 하나의 커밋에 하나의 목적만 담는다.
- 커밋 메시지 형식: `{type}: {한 줄 요약}`
  - type: feat, fix, chore, docs
- 본문에 변경 항목을 bullet으로 나열한다.
- `.env`, 크리덴셜 파일은 절대 커밋하지 않는다.

## 푸시 규칙

- main에 직접 푸시하지 않는다.
- 새 브랜치는 `-u` 플래그로 리모트 트래킹 설정한다.
- portfolio/ 폴더는 .gitignore에 포함되어 있으므로 커밋 대상이 아니다.
- **푸시 완료 후 Confluence 타임라인 페이지 업데이트** — 아래 규칙 참조.

## Confluence 타임라인 업데이트

커밋 푸시 후 아래 Confluence 페이지에 해당 날짜의 작업 내용을 추가한다:

- **페이지 ID**: `5384667150`
- **URL**: `https://findainc.atlassian.net/wiki/spaces/QAT/pages/5384667150`
- **형식**: 날짜별 내림차순 (최신이 위)
- **구조**: `<h3>YYYY-MM-DD</h3>` + `<ul><li>작업 내용</li></ul>`

**업데이트 절차**:
1. 현재 페이지 내용을 GET으로 가져온다 (`expand=body.storage,version`)
2. 기존 내용의 `<hr/>` 다음에 새 날짜 섹션을 삽입한다 (이미 해당 날짜가 있으면 항목만 추가)
3. version number를 +1하여 PUT으로 업데이트한다

**작업 항목 작성 규칙**:
- 핵심 변경만 bullet으로 (커밋 메시지 기반)
- 규칙 추가/변경은 bold 처리
- TC 생성/수정 시 건수와 우선순위 분포 표기
- **GitHub 주소를 포함하지 않는다**
